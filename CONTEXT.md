# Loom Domain Glossary

Always-loaded vocabulary for this project. Decisions live in [`DECISIONS.md`](DECISIONS.md); plans and roadmaps live in `planning/`. Update this file when a new domain term becomes load-bearing in agent or reviewer output.

## Adapter

A thin translation layer between two interfaces — e.g. a repository function that translates the domain interface into a SQL driver shape. Adapters live at module seams; they are not where business logic accumulates.

## ADR

Architecture Decision Record. Authored at `docs/adr/{NNNN}-{kebab-title}.md` with the status enum `proposed | accepted | deprecated | superseded`. Triggered only when `loom-converge` resolves a blocking conflict or `loom-roadmap converge` records a load-bearing rejection — not lazy-on-first-write.

## AgentResult

The standard TOON return envelope every agent emits. Schema lives at `protocols/agent-result.schema.md`. Carries `filesCreated[]`, `filesModified[]`, `blockers[]`, `notes[]`, `scenariosCovered[]`, and optional `contractAmendments[]`.

## Convergence Tier

The level at which a convergence target is verified. Tiers are `unit | integration | e2e | qa-review`. Each `.plan-execution/criteria-plan.toon` row carries a `tier:` and an associated runner (vitest, e2e harness, qa-review agent).

## Criteria Plan

`.plan-execution/criteria-plan.toon` — the canonical list of convergence targets a plan must satisfy. Generated by `/loom-plan create`, read by `/loom-plan test --autoconverge` and `/loom-converge --criteria`.

## Deepening

A periodic codebase-health pass run by `/loom-deepen` that surfaces shallow modules — modules with thin or low-leverage interfaces — and proposes refactoring candidates with before/after diagrams.

## Deletion Test

The vocabulary-conflict diagnostic from `protocols/codebase-design.md`: "if I removed this module/seam/adapter, what would I have to put back to make the system work?" A weak deletion test means the unit isn't load-bearing.

## Depth

How much behaviour sits behind a module's interface. Deep modules expose a small surface and carry a lot of behaviour; shallow modules expose a large surface and carry little. `/loom-deepen` flags shallow modules for refactor.

## Feedback Loop

A tight, deterministic, red-capable, agent-runnable signal that drives convergence. Schema lives at `protocols/feedback-loop.schema.md`. Carries the 10-rung escalation ladder and TRDA gate (tight, redCapable, deterministic, agentRunnable) — all four must pass before iteration begins.

## Finding

A reviewer output row. Schema lives at `protocols/findings.schema.md` with the `confidence: high | medium | low` field (default `medium`).

## Grilling

The discipline of one-question-at-a-time elicitation with a 12-question cap and `/skip` escape. Protocol lives at `protocols/grilling.md`. Five core rules (`GR-01..GR-05`) — ask one question per turn, recommend an answer, walk every branch, prefer codebase exploration over asking, cap the session.

## Handoff

A `loom-pause`-authored snapshot written to `$TMPDIR/loom-handoff-{id}.md`. Contains `suggestedSkills[]`, `referencedArtifacts[]` (paths, no duplication), and `redactedSecretsCount`. Workflow state stays in `.plan-execution/`.

## Horizontal Slicing

The anti-pattern of writing all tests first then all implementation — collapsing the red-green-refactor cycle into a single big-bang verification. The `tdd-coach` agent flags it and recommends a vertical tracer bullet instead.

## Interface

The contract a module exposes to its callers. In `protocols/codebase-design.md` framing, the interface IS the test surface — anything testable through the interface is in-scope; private internals are not.

## Kit

A bundle of typed resources (`agent | prompt | protocol | skill | infrastructure`) declared in `skills/library.yaml`. Pulled on demand via `/loom-library use <kit>`.

## Leverage

The ratio of "behaviour gained" to "interface cost" for a module. High-leverage modules expose a tiny interface and carry a lot of behaviour. Low-leverage modules are the deepening-pass target.

## Locality

Whether the consumers of a module live nearby (same package, same domain) or far (cross-cutting). High-locality modules are easier to refactor in place; low-locality modules need contract changes coordinated across consumers.

## Loop

Shorthand for `loop.toon` — a single `FeedbackLoop` envelope bound to one symptom and one command. `loom-converge` iterations bind to exactly one `loopId`; stalls escalate the LOOP down the 10-rung ladder, not the fixer.

## Module

A unit of code with a clear interface, depth, and seam boundary. Loom-native execution terms (`phase`, `wave`, `deliverable`) are NEVER used as synonyms for `module` — see the vocabulary mapping table at `protocols/codebase-design.md` Section 0.

## No-Op Test

The skill-authoring sediment-removal heuristic. For each sentence in a `SKILL.md` body: "if I deleted this sentence, would the skill still be predictable, leading-word-discoverable, and complete?" If yes, the sentence is sediment.

## Out-of-Scope Entry

A persistent rejection log row at `.out-of-scope/{id}.md`. Schema lives at `protocols/out-of-scope.schema.md`. Surfaces a visible suppression callout when `loom-roadmap converge` re-encounters a previously-rejected idea — never silent.

## Phase

A unit of plan execution. Phases sequence into waves; waves run via `/loom-plan execute`. A phase is NOT a synonym for `module` — phases are execution-pipeline structure, modules are codebase-design structure.

## Prototype

A throwaway authoring branch under `prototypes/{name}/`. Two branch types: `logic` (terminal app) or `ui` (parallel UI variants on one route). Completion ceremony writes `prototypes/{name}/answer.toon` and optionally updates the linked ADR.

## Roadmap

`planning/ROADMAP.md` — the milestone + feature graph driving plan creation. Converged via `/loom-roadmap converge` against dimension-specific reviewers.

## Seam

A planned point of substitution between modules — where one implementation can swap for another without ripple. `phasing-agent` enforces `ideal-seam-count = 1` per phase boundary: if a phase needs more than one seam, split the phase.

## Sediment

Body lines in a `SKILL.md` that no longer earn their keep — outdated examples, abandoned flags, dead branches, redundant filler. Caught by the no-op test; retired via the `scripts/sediment-sweep/no-op-test.ts` pass.

## Skill

A model-invoked or user-invoked procedural guide loaded on demand. Skills live at `skills/{name}/SKILL.md` and are registered under `skills/library.yaml`. Phase 5b's autoload-audit classifies every skill on the model-invoked vs user-invoked axis.

## State Machine

A documented set of valid + invalid transitions with explicit error codes. F-18 ships three: `FeedbackLoop` (construction → verified-red → iterating → retired), `TriageState` (needs-triage → needs-info / ready-for-agent / ready-for-human / wontfix), and `ADR` (proposed → accepted → deprecated / superseded).

## TOON

Token-Oriented Object Notation. All Loom on-disk artifacts use TOON (per `CLAUDE.md`). Exceptions are limited to app-specific data and standard tooling config files.

## Tracer Bullet

A vertical end-to-end slice exercised before broad implementation — one failing test, one minimal implementation, one refactor, then the next slice. The opposite of horizontal slicing.

## TriageState

The 5-state inbox state machine: `needs-triage | needs-info | ready-for-agent | ready-for-human | wontfix`. Schema in `protocols/out-of-scope.schema.md`; documented transitions in `commands/loom-note.md`.

## Wave

A barrier in `/loom-plan execute`. Phases group into waves; agents within a wave run in parallel; agents across waves run sequentially. A wave is NOT a synonym for `module` or `seam` — see Section 0 of `protocols/codebase-design.md`.

## Wave Gate

A required-targets bundle that must exit 0 before the orchestrator advances to the next wave. Lives in the plan's `## Wave Gates` section. Slip rules name the recovery path (or `none` when no slip is permitted).

## Wiki

`.loom/wiki/` — the project knowledge layer. Maintained by `/loom-wiki ingest`. Pre-F-18 `decision-*.md` wiki pages migrate to ADRs via `scripts/migrate-wiki-decisions-to-adrs.ts`.
